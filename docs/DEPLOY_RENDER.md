# Deploying the HR app on Render (free)

This deploys the whole app as **one free web service + one free PostgreSQL database** on
[Render](https://render.com). The single Node service serves both the API and the web app,
so you get one HTTPS URL. Everything is described by [`render.yaml`](../render.yaml).

## Deploy steps

1. **Merge to `main`** so `render.yaml` and these changes are on the default branch.
2. In Render: **New +** → **Blueprint** → connect your GitHub → pick the **`hr-app`** repo.
   Render reads `render.yaml` and shows it will create `hr-app` (web) and `hr-app-db` (Postgres).
3. When prompted for the two values, set:
   - **`ADMIN_EMAIL`** — your email (this becomes the first admin login).
   - **`ADMIN_PASSWORD`** — a temporary password you'll use once. *(Recommended to set it so
     you know it. You'll be forced to change it at first sign-in anyway. If you leave it blank,
     a random one is generated and printed in the build log.)*
4. Click **Apply**. The build will: install deps → create the DB schema → build → create your
   admin. Watch the logs for `[ensure-admin] First admin created.`
5. Open your service URL (e.g. `https://hr-app.onrender.com`, shown in Render). Sign in with
   `ADMIN_EMAIL` + the temporary password → you're **forced to set your own password**.
6. Add people: **Employees → Add employee** (or CSV import), then open each profile →
   **Create login** to issue them a unique one-time password.

## Database: Supabase Postgres (decided 2026-06-12)

Production now uses **Supabase Postgres** instead of Render's bundled Postgres, because
Render's free Postgres expires after 90 days while Supabase's free Postgres does not.
The Express + Prisma server is unchanged — only `DATABASE_URL` points elsewhere.

Project: **people-management-production** (`enbupdfxddmhgvaparaf`), region `ap-southeast-1`
(Singapore — same as the Render service), Postgres 17, free tier ($0/month).

Wiring (done 2026-06-12):

- The app connects as a **dedicated least-privilege role `peoplehub`** (LOGIN + USAGE/CREATE
  on schema `public`), not the privileged `postgres` user.
- Connection string shape (Session pooler — IPv4, port 5432; NOT the 6543 transaction
  pooler, Prisma needs session mode):
  `postgresql://peoplehub.enbupdfxddmhgvaparaf:[PASSWORD]@aws-1-ap-southeast-1.pooler.supabase.com:5432/postgres`
- The real string lives ONLY in Render → `hr-app` → **Environment → DATABASE_URL**.
  To rotate the password: run `ALTER ROLE peoplehub WITH PASSWORD '...'` in the Supabase
  SQL editor, then update the Render env var.
- The schema was pushed at setup; the runtime `db:sync + ensure-admin + seed-demo` start
  command remains idempotent on every boot.

## Keep-alive (set up 2026-06-12, $0)

A **pg_cron job inside Supabase** (`keep-hr-app-warm`, schedule `*/10 2-15 * * *` UTC ≈
07:30–21:20 IST) calls `net.http_get('https://hr-app-6qua.onrender.com/api/health')` every
10 minutes. One mechanism, two birds:

- the HTTP hit keeps the **Render free instance warm** during working hours (no cold starts);
- the cron run + the health check's DB query are **Supabase activity**, so the free project
  never hits its 7-day inactivity pause.

The window is deliberate: Render's free tier shares **750 instance-hours/month across the
workspace** (openclaw-bot lives there too); ~14 warm hours/day ≈ 420 h/month stays safely
inside it. Off-hours visits just cold-start (~30–60s). Manage the job in the Supabase SQL
editor: `select * from cron.job;` · unschedule with `select cron.unschedule('keep-hr-app-warm');`
· recent pings: `select * from net._http_response order by id desc limit 5;`

## Good to know (free tier)

- **Sleeps when idle:** covered by the keep-alive above during working hours; off-hours
  requests cold-start (~30–60s). Upgrade to a paid instance for true 24/7 always-on.
- **Data persists** in Supabase PostgreSQL across restarts and redeploys, and the free
  project does not expire (the keep-alive cron keeps it permanently active).
- **HTTPS is automatic.** `JWT_SECRET` is auto-generated by Render. Re-deploys won't reset
  accounts (`ensure-admin` only acts when no admin exists).

## Running locally after the Postgres switch

The Prisma provider is now `postgresql`, so the old `file:./dev.db` SQLite path no longer
works locally. To develop locally, either:

- **Use the included Postgres** (Docker Desktop required):
  ```powershell
  docker compose up -d
  # set server/.env: DATABASE_URL="postgresql://people:people@localhost:5432/people_management?schema=public"
  npm --prefix server run db:push
  npm run dev
  ```
- **Or point local `.env` at the Render DB** using its *External* connection string (works
  from anywhere). Good for a quick run without Docker.

> The server test suite (`npm --prefix server test`) also needs a reachable Postgres now.

## Troubleshooting

| Symptom | Fix |
|---|---|
| Build fails on `db:push` ("can't reach database") | The DB may still be provisioning on the first run — click **Manual Deploy → Deploy latest commit** to retry. |
| Didn't catch the admin password | Easiest: set `ADMIN_PASSWORD` to a known value **before the first deploy** (you change it at first login anyway). |
| App is slow on first open | That's the free-tier cold start after idle — it wakes in ~30–60s, then it's fast. |
| Want it always-on / no expiry | Upgrade the web service and Postgres to a paid plan in Render. |
